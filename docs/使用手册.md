## 二、认证方式

系统支持两种认证，任选其一。

### 2.1 Basic Auth（默认）

| 账号 | 密码 | 角色 | 权限 |
|------|------|------|------|
| `admin` | `admin` | ADMIN | 导入、管理、问答 |
| `viewer` | `viewer123` | USER | 仅问答 |

```bash
curl -u admin:admin http://localhost:8090/actuator/health
```

### 2.2 API Key（推荐脚本/CI 使用）

**配置**：在 `application.yml` 中设置密钥，然后重启应用：

```yaml
app:
  security:
    api-key: sk-my-secret-key    # 改成你自己的随机字符串
```

**使用**：用 `X-API-Key` Header 代替 `-u`：

```bash
export API_KEY="sk-my-secret-key"

curl -H "X-API-Key: $API_KEY" http://localhost:8090/actuator/health
```

> API Key 认证自动获得 ADMIN 权限。两种方式可以共存。

### 2.3 示例对照

下面所有命令给出 Basic Auth 写法，如需换 API Key，只需把 `-u admin:admin` 替换为 `-H "X-API-Key: sk-my-secret-key"`。

## 三、准备文档

把要导入的文档放到 `docs/` 目录，支持 PDF、Word、Markdown、TXT、PPT、Excel、HTML：

```bash
mkdir -p docs/api docs/database docs/product docs/wiki

echo "# 用户管理 API
## 创建用户 POST /api/users
必填参数：username（用户名）、password（密码）、email（邮箱）
可选参数：phone（手机号）、department（部门）
## 删除用户 DELETE /api/users/{id}
需要管理员权限，删除后不可恢复" > docs/api/user.md

echo "# 订单管理 API
## 创建订单 POST /api/orders
必填参数：userId（用户ID）、items（商品列表）、address（收货地址）
可选参数：couponCode（优惠券码）、remark（备注）
返回值：orderId、状态、预计送达时间" > docs/api/order.md
```

按子目录分类会自动作为 `category` 元数据（`api`、`database`、`product`、`wiki`），无子目录则归为 `other`。

## 九、常见问题

**Q: 导入后问答说"未找到相关信息"？**
- 检查 Embedding 模型是否在用：`ollama ps | grep bge-m3`
- 降低 `similarity-threshold`（`application.yml` → `app.retrieval.similarity-threshold: 0.6`）

**Q: 导入很慢？**
- 增大 `app.import.batch-size`（如 500）
- 设置 `OLLAMA_NUM_PARALLEL=4` 后重启 Ollama

**Q: Qdrant 报 UNAUTHENTICATED？**
- 确认 `application.yml` 中 `spring.ai.vectorstore.qdrant.api-key` 已配置
- 确认与服务器 Qdrant 的 API Key 一致

**Q: Prompt 报 "Missing variable names"？**
- Spring AI 1.0 GA 的 PromptTemplate 变量是 `{query}`，不是 `{question}`
- 确认 `RagConfig.java` 中模板使用 `{query}` + `{question_answer_context}`

**Q: 端口冲突？**
- 修改 `application.yml` 中 `server.port`（当前 8090）

**A: 怎么用 API Key 代替 Basic Auth？**
- 在 `application.yml` 中设 `app.security.api-key` 为非空字符串
- 然后用 `curl -H "X-API-Key: xxx"` 代替 `curl -u admin:admin`

**A: Agent 报 "does not support tools"？**
- `qwen:14b`（一代）不支持 Function Calling，需换成 `qwen3:14b`
- `ollama pull qwen3:14b`，然后改 `application.yml` 中 `chat.model: qwen3:14b`

## 十、Agent 工具开发

Agent 功能开启后（`app.agent.enabled: true`），LLM 可自主调用工具。内置工具已注册 5 个（KnowledgeTools 4 个 + DocumentSearchTool 1 个），用户可按需新增。

### 10.1 工具方法公式

```java
@Tool(name = "工具英文名", description = "给LLM看的描述——它会根据这个决定是否调用")
public String 方法名(
        @ToolParam(description = "给LLM看的参数说明") 类型 参数名) {
    // 1. 打日志
    // 2. 执行业务
    // 3. try-catch 包裹，永远返回 String
}
```

### 10.2 三要素

| 要素 | 写法 | 作用 |
|------|------|------|
| `@Tool(name, description)` | `name` 英文驼峰，`description` 中文描述 | LLM 根据 description 决定是否调用 |
| `@ToolParam(description)` | 每个参数必写 | LLM 根据 description 生成参数值 |
| `return String` | 强烈建议返回 String | LLM 直接读取内容，无需反序列化 |

### 10.3 新增工具示例

假设要加一个"订单统计"工具：

```java
import com.yt.agent.tools.AgentTool;
import org.springframework.ai.tool.annotation.Tool;
import org.springframework.ai.tool.annotation.ToolParam;
import org.springframework.stereotype.Component;

@Component
public class OrderStatsTool implements AgentTool {

    @Tool(name = "getOrderStats", description = "按日期范围查询订单统计")
    public String getOrderStats(
            @ToolParam(description = "开始日期 yyyy-MM-dd") String startDate,
            @ToolParam(description = "结束日期 yyyy-MM-dd") String endDate) {

        log.info("Agent 调用: getOrderStats({}, {})", startDate, endDate);

        try {
            // 你的业务逻辑
            int count = 156;
            return String.format("%s 至 %s 共 %d 笔订单", startDate, endDate, count);
        } catch (Exception e) {
            log.error("查询失败", e);
            return "查询失败：" + e.getMessage();
        }
    }
}
```

### 10.4 注册流程

```
新建类 implements AgentTool → 加 @Component → 写 @Tool 方法
    │
    ▼
应用启动时 Spring 扫描到所有 AgentTool 实现
    │
    ▼
AgentAutoConfiguration 自动收集 List<AgentTool>
    │
    ▼
全部注册到 ChatClient → LLM 可调用
```

> **不需要改任何现有代码。** 只需新建一个类，实现 `AgentTool` 接口，声明为 Bean。

### 10.5 参数是如何传递的（全自动）

整个过程由 Spring AI 自动完成，无需手动干预。

**第一步：启动时生成工具描述**

Spring AI 扫描 `@Tool` + `@ToolParam`，生成 JSON Schema 发给 LLM：

```json
{
  "name": "findDocuments",
  "description": "直接在知识库中搜索包含指定内容的文档片段，返回匹配的原文",
  "parameters": {
    "keyword": {
      "type": "string",
      "description": "要搜索的关键词或短语"
    },
    "topK": {
      "type": "integer",
      "description": "返回结果数量，默认 5"
    }
  }
}
```

**第二步：用户提问 → LLM 决定调用**

用户说：`"帮我在知识库里找一下支付相关的文档"`

LLM 分析过程：
1. 用户想找文档 → 匹配到 `findDocuments` 工具
2. `keyword` 从问题中提取 → `"支付"`
3. `topK` 用户没说 → 用默认值 `5`

LLM 返回指令给 Spring AI：`调用 findDocuments(keyword="支付", topK=5)`

**第三步：Spring AI 执行并反馈**

```java
findDocuments("支付", 5)  // 参数值全由 LLM 在上一步填入
  → VectorStore 检索
  → 返回原文片段给 LLM
  → LLM 拼成自然语言回答
```

**参数值谁决定的？**

| 角色 | 负责 | 示例 |
|------|------|------|
| 你（开发者） | 参数名、类型、`description` | `@ToolParam(description = "要搜索的关键词或短语")` |
| LLM | 参数**值** | 从用户问题中抽取 `"支付"` |

> **关键：`description` 写得越好，LLM 传的参数值越准。**
>
> `@ToolParam(description = "参数")` ← LLM 不知道传什么
> `@ToolParam(description = "要搜索的关键词或短语")` ← LLM 会从用户问题中提取关键词

### 10.6 注意事项

- 方法签名尽量简单——String、int 为主，不要传复杂对象
- try-catch 必须包住——工具报错会中断整个 Agent 调用链
- 返回的 String 就是 LLM 看到的——写得越清楚，回答越准确
- 放到 `com.yt.knowledge.agent` 包下，和 `DocumentSearchTool` 同级
