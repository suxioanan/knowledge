# 发布流程规范

## 版本号规则
格式：主版本号.次版本号.修订号
- 主版本号：重大架构变更，不兼容的 API 修改
- 次版本号：新增功能，向后兼容
- 修订号：Bug 修复，性能优化

## 发布检查清单
- [ ] 单元测试通过率 100%，覆盖率 > 80%
- [ ] Code Review 通过（至少 1 位 Reviewer Approve）
- [ ] 安全扫描通过（SonarQube 无 Critical/Bug 问题）
- [ ] 性能测试通过（核心接口 QPS 不低于基准值 90%）
- [ ] 数据库变更脚本已提交 Flyway 目录，且通过预发环境验证

## 发布流程
1. 创建 Release PR：合并 develop → release/vX.Y.Z
2. 预发验证：部署 staging → 冒烟测试 → 压测核心接口
3. 发布审批：运维 Leader 审批
4. 灰度发布：5% → 20% → 50% → 100%，每阶段观察 5 分钟
5. 生产验证：冒烟测试 + 核心业务指标监控

## 回滚策略
触发条件：错误率 > 1% 或 核心接口 P95 > 2x 基线
回滚方式：K8s 一键回滚到上一版本 Deployment
回滚时间：< 2 分钟
数据回滚：应用层回滚不影响数据库，如涉及数据问题走补偿脚本
